Journal de projet – Dice Roller Python

Contexte : Projet d’initiation Python, Git et Scrum dans le cadre de la formation Diginamic.

Produit : Un simulateur de lancers de dés pour Donjons & Dragons (Dice Roller), en version console dans un premier temps.

1. Objectifs du projet

  • Créer un programme Python permettant de lancer différents dés (d4, d6, d8, d10, d12, d20, d100) pour D&D.
  • Structurer le projet de manière professionnelle : arborescence src/, tests/, docs/, Git, branches…
  • Appliquer une démarche inspirée de Scrum : backlog, user stories, sprints, incréments successifs.
  • Documenter chaque étape pour en faire un livrable pédagogique et un support de démonstration de compétences.

2. Objectifs pédagogiques

  • Git / GitHub : initialisation de dépôt, arborescence, commits, branches (sprints), README.
  • Python : organisation du code, séparation logique / interface (core vs CLI), fonctions et modules.
  • Gestion de projet : formaliser un cahier des charges fonctionnel, un backlog, des user stories.
  • Documentation : produire des livrables clairs, traçables, publiables (HTML / Markdown).

3. Étapes du projet (timeline)

Étape 1 – Initialisation du dépôt Git et création de l’arborescence

Objectif : Créer la base du projet avec une structure claire et l’initialiser sous Git.

Actions réalisées :

  • Choix du nom de projet (ex. dice-roller-python).
  • Création du dossier du projet et initialisation du dépôt Git.
  • Création d’une arborescence standard de projet :
mon-projet/
 ├─ src/          # code source
 │   ├─ main.py
 │   └─ modules/
 │
 ├─ tests/        # tests unitaires
 ├─ docs/         # documentation
 ├─ README.md
 ├─ .gitignore
 └─ .github/

Exemple de commandes (Git Bash) :

# Vérifier où l’on se trouve
pwd
ls

# Nom du projet
REPO_NAME="dice-roller-python"

# Création du dossier et entrée dans le dossier
mkdir "$REPO_NAME"
cd "$REPO_NAME"

# Initialiser le dépôt Git
git init

# Créer l’arborescence minimale
mkdir -p src/modules
mkdir -p tests
mkdir -p docs
mkdir -p .github

# Fichiers de base
touch src/main.py
touch src/modules/__init__.py
touch tests/test_placeholder.py
touch docs/notes.md
touch README.md
touch .gitignore

Compétences mises en avant :

  • Organisation d’un projet logiciel (structure src/tests/docs).
  • Maîtrise des bases de Git (init, suivi de fichiers).
  • Utilisation de Git Bash pour automatiser la création de l’arborescence.

Étape 2 – Création et remplissage du README

Objectif : Documenter dès le départ le but du projet et sa structure.

Actions réalisées :

  • Création d’un fichier README.md ou README_[nom_du_dossier].txt.
  • Ajout d’une description courte du projet.
  • Liste des dossiers principaux et de leur rôle.

Exemple de contenu :

# dice-roller-python

Projet Dice Roller - Simulateur de jets de dés pour Donjons & Dragons (D&D) en Python.

Structure du projet :
- src/ : code source de l'application (main.py, modules, etc.)
- tests/ : tests unitaires
- docs/ : documentation (cahier des charges, backlog, journal de projet, etc.)
- .github/ : configuration GitHub (workflows, CI, etc.)

Compétences mises en avant :

  • Capacité à présenter un projet de façon synthétique.
  • Organisation de la documentation dès le début du projet.

Étape 3 – Cahier des charges fonctionnel (CdCF) Dice Roller

Objectif : Définir clairement ce que doit faire le module Dice Roller.

Actions réalisées :

  • Rédaction d’un cahier des charges fonctionnel en HTML (ou Markdown) décrivant :
    • Le contexte : utilisation pendant des parties de D&D en ligne.
    • Les dés gérés : d4, d6, d8, d10, d12, d20, d100 (avec d100 simulé via 2d10 possible).
    • Les fonctionnalités : lancer un ou plusieurs dés, afficher les résultats, calculer une somme.
    • Les cas particuliers : échec critique (1) et réussite critique (20) sur les d20.
    • Les modificateurs (+X / -X) à appliquer plus tard sur les résultats.
  • Positionnement de ce CdCF dans un dossier docs/ dédié à la documentation.

Compétences mises en avant :

  • Analyse des besoins à partir d’un usage réel (D&D, Baldur’s Gate 3, avantage/désavantage, etc.).
  • Rédaction d’un cahier des charges fonctionnel structuré.
  • Préparation du terrain pour la création de user stories (Scrum).

Étape 4 – Backlog produit et user stories (V1 console)

Objectif : Transformer le CdCF en backlog exploitable dans une démarche Scrum.

Actions envisagées / en cours :

  • Création d’un fichier product-backlog dans docs/.
  • Rédaction des user stories principales pour la V1 console, par exemple :

Exemples de user stories :

US-001 - Lancer un dé simple
En tant que joueur de D&D,
Je veux lancer un dé avec un nombre de faces donné (d4, d6, d8, d10, d12, d20, d100),
Afin de remplacer les lancers physiques quand je joue en ligne.

US-002 - Lancer plusieurs dés
En tant que joueur de D&D,
Je veux lancer plusieurs dés du même type (par exemple 2d20),
Afin de gérer l'avantage, le désavantage ou plusieurs jets groupés.

Compétences mises en avant :

  • Application de la méthode Scrum (backlog, user stories, priorisation).
  • Capacité à faire le lien entre CdCF et backlog produit.

Étape 5 – Implémentation V1 (squelette Python)

Objectif : Mettre en place la structure minimale du code pour la V1 console.

Actions envisagées / en cours :

  • Créer une fonction principale dans src/main.py pour lancer le programme.
  • Préparer un module dédié au cœur du lancer de dés (par exemple src/modules/dice_roller.py).
  • Prévoir une future séparation entre :
    • la logique métier (calcul des jets, critiques, modificateurs),
    • et l’interface (ligne de commande, puis plus tard interface graphique).

Compétences mises en avant :

  • Structuration du code Python (modules, fonctions, point d’entrée).
  • Préparation à une évolution vers des versions futures (GUI, réseau, etc.).

Étape 6 – Commit et traçabilité des étapes

Objectif : Assurer une bonne traçabilité du projet dans Git.

Actions envisagées / en cours :

  • Après chaque grande étape (arborescence, CdCF, backlog, squelette Python), effectuer :
git status
git add .
git commit -m "Description claire de l'étape (ex : init arbo, ajout CdCF, etc.)"
  • Utiliser des messages de commit explicites pour raconter l’histoire du projet.
  • Éventuellement, créer des branches par sprint ou par version (ex. sprint/01-v1-console).

Compétences mises en avant :

  • Bonne pratique Git (commits réguliers, messages explicites).
  • Capacité à relier chaque commit à une étape du journal de projet.
